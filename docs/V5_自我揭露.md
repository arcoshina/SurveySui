# V4 受眾篩選（Eligibility）定稿規格

> 狀態：**定稿（v1 留空實作中）**  
> 相關：`survey_registry`、`survey_eligibility::claim_v2`、`survey_pass::SRC_ATTRIBUTES`

## 1. 設計原則

| 項目 | 決策 |
|------|------|
| Merkle / schema_version | **不做** |
| `attributes_nullifier`（整包自填壓成一個 hash） | **不做** |
| BFF 參與 Stage1→Stage2 | **不做**；BFF 仍只處理 Email/OAuth/World ID **Mint ticket** |
| 自填真偽驗證 | **不做**；合約只算「命中幾個 allowlist 項」 |
| 產品目標 | 在隱私法規內**提高臨時改資格的配合成本**（如忽男忽女），非鏈上驗真 |
| Vault 重複領獎去重 | **不變**：`survey_pass::all_nullifiers` → `H(nullifier‖vault_id)` |
| 問卷 eligibility 欄位 | **register 後不可改** |
| 新入口 | **`survey_eligibility::claim_v2`**（package 升級用新 module） |

## 2. 兩階段流程（產品語意）

1. **Stage1（資格問卷）**：前端依各維度對自填答案做 hash，**只留在客戶端**。
2. **Stage2（主問卷）**：前端把 `attribute_nullifiers: vector<vector<u8>>` 當 **交易輸入** 傳入 `claim_v2`；**不寫入 Pass**、**不經 BFF**。

v1 合約：**不強制** Stage1 已完成（`stage1_survey_id` 僅儲存）。

## 3. `Survey` 擴充欄位

| 欄位 | 型別 | v1 行為 |
|------|------|---------|
| `allowed_nullifiers` | `vector<vector<u8>>` | 空 → **略過**受眾檢查 |
| `match_threshold` | `u64` | 預設 `0`；allowlist 非空時需 `hits >= threshold` |
| `disclosure_rule_blob` | `Option<vector<u8>>` | **忽略**（留空給未來規則引擎） |
| `stage1_survey_id` | `Option<ID>` | 僅儲存；v1 不 enforce |
| `claim_mode` | `u8` | `0` = Pass + 受眾輸入；`1` = 一次性 ticket（**保留**，v1 `abort`） |

既有 `allowed_sources` 可含 **`SRC_ATTRIBUTES = 8`**：僅表示「可走受眾比對路徑」，**無 tier**、**無 CredentialSlot**。

## 4. `claim_v2` 邏輯（v1）

```
1. 標準 claim 前置：vault 開啟、未過期、survey↔vault 綁定、pass 有效、sender = pass.owner
2. claim_mode == 1 → abort（未實作）
3. 資格（OR，與現有 check_eligibility 一致）：
   - 任一非 SRC_ATTRIBUTES 的 allowed_source 在 Pass 上有效；或
   - allowed_sources 含 SRC_ATTRIBUTES 且受眾檢查通過
4. **且** 受眾檢查（僅當 `allowed_nullifiers` 非空）：
   hits = |attribute_nullifiers ∩ allowed_nullifiers|
   assert hits >= match_threshold
5. Vault nullifier 去重 + 發獎（與 claim 相同）

步驟 3 與 4 為 **AND**：有設定 allowlist 時，即使 Pass 憑證有效仍須命中受眾。
```

`attribute_nullifiers` **只用於受眾命中**，**不用於** vault 去重。

## 5. `SRC_ATTRIBUTES`（source = 8）

- 常數定義於 `survey_pass.move`。
- **不在** tier 映射中。
- `survey_vault::check_eligibility`（舊 `claim`）會 **跳過** source 8，避免誤判 slot。
- 僅含 `[8]` 的問卷：舊 `claim` **無法**領獎；需 `claim_v2` + 受眾路徑。

## 5.1 雙入口（上線篩選前必封）

受眾比對**只掛在** `survey_eligibility::claim_v2`；舊的 `survey_vault::claim` **不讀** `allowed_nullifiers`。

若問卷已設非空 allowlist，填答仍走舊 `claim`，等於「臨時改資格」這層成本**被繞過**（Pass 憑證過即可領獎）。

**上線篩選前**須封雙入口，至少其一：

- 合約：`allowed_nullifiers` 非空時舊 `claim` **abort**；或
- 前端 + BFF gas sponsor：**只允許** `claim_v2`。

v1 預設 allowlist 為空、產品仍走舊 `claim`，此風險**尚未觸發**。

### 6.3 開放問題（實作前需定案）

1. **Hash 公式**：各維度 `H(dimension_id ‖ normalized_value ‖ survey_salt?)` 的字節序與 salt 來源。
2. **allowlist 上限**：v1 建議 `MAX_ALLOWED_NULLIFIERS = 100`（gas 與 tx 大小）。
3. **OR vs AND**：`[Email, SRC_ATTRIBUTES]` 目前為 **OR**；若產品要「必須 Email 且命中受眾」，需另訂 `claim_mode` 或拆分欄位。

## 7. 預設值（向後相容）

新問卷若未設定篩選，register 應傳：

- `allowed_nullifiers = []`
- `match_threshold = 0`
- `disclosure_rule_blob = none`
- `stage1_survey_id = none`
- `claim_mode = 0`

行為等同現有 `survey_vault::claim`（無受眾門檻）。

## 8. 測試計畫（v1）

| `allowed_sources = [8]`、空 allowlist、`claim_v2` | 過（受眾路徑 placeholder） |
