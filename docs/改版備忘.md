# MVP 完成後再改版的目標  

## 問卷發起流程 (發起者)  

- 調整總手續費到 20%，可更新 <100% 的折扣比率，預設 50%  
- 設定獎勵規則與參與條件：寫在 .md 文件的檔頭當作 metadata  
- 發起問卷流程：
   1. 預估「總 Token 消耗」+「平台手續費」（到 20%）。  
   2. 發起者輸入要投入的`stackedSurveySuiReward` 和 `Sui` 的數量  
   3. 先建立金庫 `SurveyVault`  
   4. 將 將發起者投入的 Sui 注資到 AMM 池中，池中 Sui 增加並鑄造 `SurveySuiReward` (SSR) 作為儲備留在池內，同時 1:1 鑄造等額的 `stackedSurveySuiReward` (sSSR)  
   5. 兩包 sSSR 合併，依費率分出手續費和獎勵，手續費傳給管理者，獎勵 sSSR 存入金庫。**新的 sSSR 不會傳給發起者**  
   6. 確認額度足夠後，註冊問卷  

## 問卷檢查  

- 先檢查重複，再檢查內容  
- 不僅檢查非空，要檢查完整問卷(前端內容可能被竄改)  
- 考慮 gas 消耗，尋找合適的替代方案，包含但不限於 TEE  

## 確認事項  

- 如果後端失效，是否能先 dryrun 檢查再代付  
- SurveyPass 要在受訪者首次連錢包時檢查及提示  
- 任何檢查都不能只依靠前端，因為前端內容可以被竄改  
- 問卷填答結果格式? MD? CSV? 其他?  
- 發起人的加密用公鑰用物件直接存發起人錢包，考慮是否寫進 Pass  
- 用戶領出SSR後，能不能存回 pool ，領取 sSSR  
- 在devnet上沒差，但如果在其他網路，可以用同一組註記詞產生幾組地址來測試，方便重複使用。
