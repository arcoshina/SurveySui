## 一、各國電子投票安全標準

### 1.1 歐洲理事會準則 CM/Rec(2017)5

歐洲理事會 2017 年更新的電子投票建議標準，是目前最具權威的電子投票框架，被愛沙尼亞、瑞士、挪威等國採用。核心安全性質：

| 性質 | 定義 | SurveySui 對應難點 |
|------|------|-------------------|
| **投票資格**（Eligibility） | 只有合格選民可投票 | SurveyPass tier 門檻可處理 |
| **唯一性**（Uniqueness） | 每人只能投一票 | nullifier 可處理 |
| **匿名性**（Anonymity） | 選票無法追溯到投票者身份 | 需要 ZKP + relayer |
| **個人可驗性**（Individual verifiability） | 投票者可確認自己的票被正確計入 | 需要 receipt + 鏈上 commitment |
| **全局可驗性**（Universal verifiability） | 任何人可驗證計票結果正確 | 鏈上透明可達成 |
| **抗脅迫性**（Coercion resistance） | 無法向第三方證明自己投了什麼 | 需要 re-vote 機制或 fake credential |
| **無收據性**（Receipt-freeness） | 無法產生可信的投票收據 | 最難達成的性質 |
| **完整性**（Integrity） | 選票不被竄改 | 鏈上不可變性可達成 |
| **可稽核性**（Auditability） | 系統可接受獨立稽核 | 開源合約 + 鏈上資料可達成 |

### 1.2 愛沙尼亞 i-voting 系統（最接近 Web3 的實踐）

愛沙尼亞是全球唯一在國家選舉中使用線上投票（i-voting）的民主國家，已運行超過 20 年。

**關鍵架構特性**：
- **身份驗證**：國家 PKI 智慧卡（eID）或行動 ID；嚴格一人一票
- **投票匿名化**：身份驗證與選票提交在架構上分離——身份伺服器和計票伺服器各自持有不同密鑰，無法聯合還原「誰投了誰」
- **選票加密**：ElGamal + 國家選委會 threshold 公鑰，選舉結束前無法解密任何單張選票
- **Re-vote（換票機制）**：投票期間可重複投票，只有最後一張有效——這是達成抗脅迫性的核心設計
- **個人可驗性**：投票後可在截止前驗證選票已被收錄（不洩漏內容）
- **開源稽核**：程式碼公開，選後可下載所有加密選票自行驗算

### 1.3 美國 VVSG 2.0（2021）

美國選舉協助委員會（EAC）的自願性投票系統指引，重點在軟體系統審計與投票紀錄保存，較 CoE 標準更強調物理安全，對純線上投票持保守態度。SurveySui 可參考其「可稽核性」要求，不強求其物理安全條款。

### 1.4 SurveySui 的合規定位

SurveySui 不是政府投票系統，但「符合嚴格要求」意味著：

| 標準 | V3 目標 | V4+ 目標 |
|------|---------|---------|
| 投票資格 | ✓ SurveyPass tier 門檻 | ✓ |
| 唯一性 | ✓ ZK nullifier | ✓ |
| 匿名性 | ✓ sender 與 answer 不連結 | ✓ |
| 個人可驗性 | ✓ 鏈上 commitment | ✓ |
| 全局可驗性 | ✓ 鏈上透明計票 | ✓ |
| **抗脅迫性** | ✗ 不支援 | ✓ Re-vote 機制 |
| 可稽核性 | ✓ 開源合約 | ✓ |

## 二、威脅模型

### 2.1 攻擊者分類

| 攻擊者 | 能力 | 防禦要求 |
|--------|------|---------|
| **網路竊聽者** | 監聽封包、IP 關聯 | HTTPS + Relayer（Gas Station）隱藏 IP |
| **Gas Station 操作者** | 看到 TX 內容（不含加密票） | 匿名 relayer；不存 IP 日誌 |
| **BFF 操作者** | 知道誰驗過身份 | 架構分離：BFF 不知道投票內容 |
| **合約資料分析者** | 讀取全部鏈上 TX | ZKP 使 nullifier 無法追溯身份 |
| **脅迫者** | 要求投票者「證明」投了誰 | Re-vote 機制（V4+） |
| **共謀攻擊**（BFF + Gas Station） | 交叉比對時間戳 | 時間混淆 + 批次提交（V4+） |

### 2.2 攻擊面矩陣

```
                投票資格     唯一性     匿名性     完整性
BFF 漏洞          ▲▲         ▲          ▲          -
Gas Station         -          -         ▲▲          -
鏈上分析            -          -          ▲          -
客戶端             ▲▲         -         ▲▲          -
時間相關分析         -          -          ▲          -
共謀攻擊           -          -         ▲▲          -

▲▲ = 主要威脅   ▲ = 次要威脅
```

### 2.3 不在防禦範圍內

- **客戶端被植入惡意程式**：若瀏覽器被控制，任何 e-voting 系統均無效
- **投票者自願洩漏**：投票者主動分享自己的投票 = 信任假設破壞
- **SurveyPass 本身的 Sybil 攻擊**：此屬 SurveyPass 防女巫機制的責任範圍

## 三、密碼學方案比較

| 方案 | 匿名性 | 可驗性 | 抗脅迫 | 鏈上成本 | Sui 相容性 | 建議用途 |
|------|--------|--------|--------|---------|-----------|---------|
| **Nullifier-only**（無 ZKP） | ✗ 地址可見 | ✓ | ✗ | 低 | ✓ | 不適用 |
| **Blind signature + Relayer** | △ BFF 知時間 | △ | ✗ | 低 | ✓ | Phase 1 過渡 |
| **Semaphore ZKP**（Groth16） | ✓✓ | ✓ | △ | 中 | ✓（Move groth16） | **推薦主架構** |
| **Mix-net** | ✓✓ | ✓ | △ | 高 | △ 需多方 | 可與 ZKP 互補 |
| **Homomorphic encryption** | ✓ 對票數 | ✓ | ✗ | 高 | △ 無原生支援 | 隱私計票輔助 |
| **PLONK/STARK** | ✓✓ | ✓ | △ | 極高 | ✗ 尚無移植 | 未來研究 |

**選擇依據**：
- Sui Move 內建 `sui::groth16` 模組，可鏈上驗 Groth16 proof（World ID / Semaphore 用同一底層）
- Semaphore 協議已被 Ethereum 生態廣泛驗證（1M+ 用戶的 World ID），電路設計成熟
- Nullifier 結構與 SurveyPass 現有 commitment 架構相容，修改範圍可控

## 四、推薦架構：Semaphore-style ZKP

採用與 World ID（Semaphore）相同的密碼學原語，利用 Sui 原生 Groth16 驗證器，達到鏈上可驗的匿名投票。

### 4.1 核心組件

```
[客戶端]                [Gas Station]          [Sui 鏈上]
  投票者                  匿名 Relayer
  
  vote_secret (s)                            CommitmentTree
  ← 衍生自錢包私鑰                             ← shared object
                                               └ root: Field252
  
  proof 生成                                 NullifierSet
  (WASM circuit)                              ← shared object
                                               └ used: Table<Field252, bool>
  
  encrypted_vote                             VaultAnswers
  ← 用問卷發起者 pubkey                        ← 已有結構
    加密（X25519 + AES）
```

### 4.2 Nullifier 結構

```
身份層（SurveyPass 發行時建立）
  vote_secret   s  = HMAC-SHA256(wallet_privkey, "surveysui-vote-v1")
                     ← signPersonalMessage("surveysui-vote-v1") 衍生
  salt          r  = random_bytes(32)
  commitment    C  = Poseidon2(s, r)
               ↑ 存入 CommitmentTree，鏈上可見，無法還原 s

投票層（每次投票計算）
  vault_nullifier N = Poseidon2(s, vault_id)
                      ← 相同 (s, vault_id) → 相同 N（防重複）
                      ← 不同 vault_id → 不同 N（防跨問卷追蹤）
                      ← 不可從 N 還原 s（單向）

ZK Proof 聲明：
  「我知道 (s, r, merkle_path) 使得：
    1. Poseidon2(s, r) 是 CommitmentTree root 下的合法葉節點
    2. Poseidon2(s, vault_id) = N（我提交的 nullifier）」
```

**Poseidon2 選擇理由**：
- ZK-circuit 友善的 hash function（比 SHA256 便宜 100×）
- BN254 曲線原生支援（Groth16 用同一曲線）
- 已有成熟 Rust/WASM 實作（circomlib / iden3）

### 4.3 CommitmentTree（累加型 Merkle Tree）

```
深度：20（支援最多 2^20 ≈ 100萬張 Pass）
Hash：Poseidon2
更新：每次 mint_pass 時，Pass owner 可選擇性呼叫 register_vote_commitment(pass, C)
       → 在合約內將 C 加入 next leaf 位置，更新 Merkle root

Proof 生成：
  ← BFF（或客戶端完整下載樹）提供 Merkle inclusion proof
  ← 客戶端 WASM circuit 生成 Groth16 proof（3~10 秒，不上傳私鑰）
```

**重要設計決策**：`register_vote_commitment` 與 `mint_pass` 分開
- 理由：不是所有 Pass holder 都需要匿名投票（tier 0/1 可能不允許）
- 好處：CommitmentTree 不會因為大量非投票 Pass 而膨脹

### 4.4 協議流程（三階段）

#### 階段一：Pass 發行後的一次性設定

```
1. 用戶持有有效 SurveyPass（任何 tier）
2. 前端提示：「此問卷支援匿名投票，建議設定投票憑證」
3. 用戶點擊 → signPersonalMessage("surveysui-vote-v1")
   ← 確定性衍生 vote_secret s（不存儲，每次重新衍生）
4. 客戶端生成隨機 salt r，計算 C = Poseidon2(s, r)
5. r 加密存於 localStorage（以 s 為密鑰）
6. 呼叫 register_vote_commitment(pass, C)
   ← 合約：verify pass active + not already registered → insert C to tree
```

#### 階段二：匿名投票

```
1. 用戶在 SurveyPage 選擇「匿名投票」模式
2. 前端：signPersonalMessage("surveysui-vote-v1") → 衍生 s
         讀取 localStorage 中加密的 r → 解密
3. 計算 nullifier N = Poseidon2(s, vault_id)
4. 加密投票內容：encrypted_vote = X25519-XSalsa20Poly1305(vote, creator_pubkey)
5. 從 BFF 取得 CommitmentTree 的 Merkle inclusion proof（for C）
6. 客戶端 WASM 生成 Groth16 proof π（離線，不傳送任何私鑰）
7. 通過 Gas Station 提交：submit_anonymous_vote(N, encrypted_vote, π, tree_root)
   ← Gas Station 作為 TX sender，隱藏投票者地址
8. 合約驗證：
   a. 驗 Groth16 proof（N, tree_root, π）→ 有效則繼續
   b. 檢查 NullifierSet：N 未使用 → 記錄 N = true
   c. encrypted_vote 存入 VaultAnswers
```

#### 階段三：解密計票

```
問卷關閉後：
1. 問卷創建者使用 S4.1 設定的 decryption key 解密所有 encrypted_vote
   ← 此密鑰只有創建者持有，BFF 不知道
2. 統計時：只看 (nullifier, vote_index) 對應關係
   ← nullifier 不可追溯到 pass_owner
3. 如需公開驗算：發佈所有 encrypted_vote + 解密 proof（不含密鑰）
```

### 4.5 答案加密策略

| 問卷類型 | 加密方式 | 誰能解密 |
|---------|---------|---------|
| 公開問卷 | 明文（現有架構） | 任何人 |
| 創建者私密 | X25519 + 創建者 pubkey | 只有創建者 |
| 未來：閾值解密 | ElGamal + threshold (t-of-n) | 至少 t 個委員合作 |

## 五、與 SurveyPass 防女巫的衝突解法

### 5.1 衝突點分析

| 維度 | SurveyPass 防女巫（現有） | 匿名投票（新增） |
|------|------------------------|----------------|
| **驗證方式** | Pass holder 即 TX sender | TX sender = Gas Station（relayer） |
| **nullifier 用途** | 防止同一 email 拿多張 Pass | 防止同一 Pass 在同一問卷重複投票 |
| **commitment 用途** | 屬性 Merkle root（KYC） | 身份匿名集合（CommitmentTree） |
| **關聯性** | Pass 綁定 address（可見） | 投票與 address 解耦 |

**核心衝突**：SurveyPass 「持有即驗證（Pass holder = sender）」模型，與匿名投票「sender ≠ voter」需求直接衝突。

### 5.2 解法：雙 nullifier 模型

**不修改** SurveyPass 的防女巫 nullifier（`issuance_nullifier = hash("email" || email)`）。
**另外建立** voting nullifier 系統，兩者獨立運作：

```
Issuance nullifier：   用於 NullifierRegistry（防止同一真人多張 Pass）
                       hash("email" || email_address) → 不可用於投票追蹤

Vote commitment：       用於 CommitmentTree（匿名集合）
                       Poseidon2(vote_secret, salt)

Vote nullifier：        用於 VaultNullifierSet（防重複投票）
                       Poseidon2(vote_secret, vault_id)
```

三個 nullifier 互相獨立，破解任一個無法推出其他兩個。

### 5.3 Pass struct 修改方案

在 `SurveyPass` 新增一個欄位：

```move
struct SurveyPass has key {
    // ... 既有欄位 ...
    
    // 新增（S5.2 設計，實作在 V3）
    vote_commitment: Option<vector<u8>>,  // None = 未設定投票憑證
}
```

在 `CommitmentTree`（新增 shared object）：

```move
struct CommitmentTree has key {
    id:          UID,
    root:        vector<u8>,   // current Merkle root（BN254 Field252）
    next_index:  u64,
    leaves:      vector<vector<u8>>,  // 合約僅存 hash，完整樹由 BFF 維護
}
```

**向後相容**：`vote_commitment = None` 時，匿名投票功能不啟用，Pass 現有功能不受影響。

## 六、合規差距分析（CoE CM/Rec(2017)5）

| 性質 | V3 達成程度 | 差距說明 | 彌補路徑 |
|------|-----------|---------|---------|
| 投票資格 | ✓ | — | — |
| 唯一性 | ✓ | nullifier 防重複 | — |
| 匿名性 | ✓ | ZKP + relayer | 補強 relayer IP 匿名（V4） |
| 個人可驗性 | ✓ | 鏈上 commitment | — |
| 全局可驗性 | ✓ | 鏈上透明 | — |

| 可稽核性 | ✓ | 開源 + 鏈上 | — |

**抗脅迫性的根本限制**：任何客戶端執行的 e-voting 系統（包括愛沙尼亞）都面臨此問題。愛沙尼亞的解法是 Re-vote：脅迫者無法確定最後一票是否被改掉。SurveySui V4 採用相同策略。

## 七、預估工作量

### Phase 1（V3 目標）：基礎匿名投票

**範圍**：達成「匿名性 + 唯一性 + 個人/全局可驗性」，不含抗脅迫性。

| 工項 | 說明 | 預估工時 |
|------|------|---------|
| Groth16 電路設計 | Semaphore-compatible circuit（Poseidon2 + Merkle inclusion + nullifier derivation） | 3–5 週 |
| WASM proof 生成 | 移植 circomlib Poseidon + ptau 設定儀式 | 2–3 週 |
| Move 合約 | CommitmentTree、VaultNullifierSet、submit_anonymous_vote entry | 2–3 週 |
| BFF 更新 | CommitmentTree 快照 API、Merkle proof 生成 | 1–2 週 |
| 前端 | 匿名模式 UX 切換、signPersonalMessage 衍生、WASM 整合 | 2–3 週 |
| 測試 | Move test_scenario + Frontend ZK integration | 2 週 |
| **合計** | | **12–18 週** |

**關鍵前置工作**：
1. Groth16 verifier key 生成（Powers of Tau 設定儀式，社群參與為佳）
2. Move `sui::groth16` API 確認可支援 Semaphore circuit 的 vk 大小
3. Gas Station 確認匿名 relayer 模式（不記錄 IP）

### Phase 2（V4 目標）：抗脅迫性

**範圍**：新增 Re-vote 機制，使脅迫無法確認最終投票。

| 工項 | 說明 | 預估工時 |
|------|------|---------|
| Re-vote 合約 | NullifierSet → NullifierMap（nullifier → slot_index）；submit_anonymous_vote 支援覆蓋 | 1–2 週 |
| 投票截止機制 | VaultAnswers 加 voting_ends_at，截止後鎖定 | 0.5 週 |
| 前端 Re-vote UX | 顯示「已投票，可在截止前更換」 | 1 週 |
| **合計** | | **2.5–3.5 週** |

## 八、下一版（V3）啟動條件

以下條件**全部達成**才啟動 V3 匿名投票實作：

### 技術前置條件

- [ ] **T1** `sui::groth16` 模組確認可驗 Semaphore-compatible Groth16 proof（實際測試，非文件聲稱）
- [ ] **T2** Poseidon2（BN254）有可用的 Rust crate 或 circomlib circuit，且已有 npm/WASM 發布版
- [ ] **T3** 完成簡易 proof-of-concept：在 Sui Devnet 上驗一條 Semaphore-style proof，Gas 消耗 < 1 SUI
- [ ] **T4** CommitmentTree 深度 20 的 Merkle inclusion proof 生成時間 < 10 秒（客戶端）

### 設計前置條件

- [ ] **D1** SurveyPass tier 門檻要求定案：匿名投票限定哪些 tier？（建議 tier ≥ 1，Email OTP 以上）
- [ ] **D2** CommitmentTree 初始化策略定案：是否在問卷創建時建立獨立 tree，或全域共享一棵
- [ ] **D3** 問卷創建者 decryption key 管理策略定案（S4.1 兩步驟金鑰是否直接沿用）
- [ ] **D4** Gas Station 匿名 relayer 政策定案（不記錄 IP vs. 有限日誌 vs. 完全匿名）

### 產品前置條件

- [ ] **P1** 至少 2 個問卷創建者的實際需求確認：「我需要匿名投票功能」
- [ ] **P2** 評估是否需要 Powers of Tau 設定儀式（公信力 vs. 工程成本）
- [ ] **P3** 法律評估：匿名投票的投票結果是否具備 SurveySui 服務條款層面的法律效力聲明

## 附錄 B：術語表

| 術語 | 說明 |
|------|------|
| **Commitment** | `C = Poseidon2(s, r)`；公開可見，無法還原 s |

| **Groth16** | 一種 ZK-SNARK 方案，proof 大小固定（≈ 128 bytes），驗證快 |
| **Poseidon2** | ZK-circuit 友善的 hash 函數，BN254 曲線原生支援 |
| **Re-vote** | 投票期間允許覆蓋舊投票；抗脅迫的核心機制 |
| **Relayer** | 代替投票者提交 TX 的中介方（Gas Station 角色） |
