# SurveySui 上線部署指南

本文件提供將 `SurveySui` 專案發布至正式生產環境（Mainnet）或公開測試網（Testnet）的完整標準作業流程。  
在開始部署前，請確保已詳閱 [安全指引.md](./安全指引.md) 與 [託管架構.md](./託管架構.md)，並嚴格遵循金鑰隔離與最小權限原則。

---

## 1. 部署架構與前置準備

### 1.1 部署架構概覽

```mermaid
flowchart TD
  subgraph Client [前端託管]
    CFPages[Cloudflare Pages 前端網頁]
  end

  subgraph Server [後端服務]
    BFF[BFF API 服務 Render / Fly.io / VPS]
    CFWorker[Cloudflare Gas Station Worker]
    CFD1[(Cloudflare D1 SQLite)]
  end

  subgraph Blockchain [Sui 區塊鏈]
    SuiChain[Sui Mainnet / Testnet]
  end

  CFPages -->|API 請求| BFF
  BFF -->|代付轉發 (DO)| CFWorker
  CFWorker -->|2-of-3 多簽簽章| SuiChain
  CFWorker -->|狀態記錄| CFD1
  BFF -->|驗證 Ticket| SuiChain
```

### 1.2 金鑰與權限準備（極重要）
在部署前，您需要準備並區分以下金鑰，**嚴格禁止混用**：

| 金鑰變數 | 層級 (Tier) | 用途 | 部署去向 | 安全規範 |
| :--- | :--- | :--- | :--- | :--- |
| `SUI_ADMIN_PRIVATE_KEY` | **Tier 0** | 部署合約、初始化 AMM 資金池與合約升級。 | **僅保留於本機部署終端**。 | **絕對不可**設定於 BFF 運行環境，BFF 偵測到此 Key 會直接 Crash 拒絕啟動。 |
| `SURVEY_PASS_ISSUER_PRIV` | **Tier 1** | 簽發 SurveyPass 准考證 Ticket (Ed25519)。 | BFF 運行環境。 | 專用於 Ticket 簽章，**不可**作為 Gas 代付帳戶。 |
| `GAS_SPONSOR_PRIV_1` / `_2` | **Tier 1** | 2-of-3 多簽代付帳戶之**熱簽名私鑰** (K1/K2)。 | BFF 與 Cloudflare Worker 運行環境。 | 用於全自動代付簽章，若單一熱鍵外洩，資金仍安全。 |
| `GAS_SPONSOR_PUBKEY_3` | **Tier 1** | 2-of-3 多簽代付之**冷備份公鑰** (K3 公鑰)。 | BFF 與 Cloudflare Worker 運行環境。 | 用於與 K1/K2 公鑰組合成多簽地址。 |
| `GAS_SPONSOR_PRIV_3_OFFLINE` | **Tier 1** | 2-of-3 多簽代付之**冷備份私鑰** (K3 私鑰)。 | **僅線下安全儲存**（如密碼管理器、紙本、冷錢包）。 | **嚴禁**部署至任何雲端或伺服器運行環境。 |

## 2. 第一階段：Sui Move 智能合約部署

本步驟將編譯 Move 合約、發行 package 至鏈上，並完成 `Currency` 註冊與 AMM 資金池初始化。

### 步驟 2.1：設定部署用環境變數
在 repo 根目錄建立（或修改）`.env` 檔案。填入以下**部署專用**變數：

```env
# ── Tier 0：合約管理（僅部署腳本使用） ──
SUI_ADMIN_PRIVATE_KEY=suiprivkey1... # 您的合約 Admin 私鑰
SUI_ADMIN_ADDRESS=0x...            # 您的合約 Admin 地址

# ── 網路與 RPC 設定 ──
SUI_NETWORK=mainnet                 # 可選：mainnet / testnet / devnet
SUI_RPC_URL=https://fullnode.mainnet.sui.io:443 # 對應網路的 RPC Node URL

# ── 營運初始參數 ──
MIN_GAS_COMPENSATION_AMOUNT=100000000 # 最低代付補償 MIST (預設 0.1 SUI)
PURGE_ANSWERS_BATCH=100               # 銷毀答卷批次大小
SUI_PURGE_SPONSORS=0x...              # 授權於正常寬限期(92天)清除問卷的 sponsor 地址，逗號分隔，上限 3
```

> [!IMPORTANT]
> `SUI_PURGE_SPONSORS` 應填入 **BFF 多簽代付地址 `GAS_SPONSOR_ADDRESS`**（步驟 3.1 產生），這是 BFF 簽署 `purge` 交易所用的身分。
> 合約 `purge` 權限分三層：**問卷發起者**（關閉後可立即銷毀）、**`SUI_PURGE_SPONSORS` 名單 / 合約 admin**（到期 + 92 天寬限期後可銷毀，BFF 自動清理走此路徑）、**其他任何人**（僅作為 liveness 備援，需到期 + 365 天後）。
> **順序提醒**：本指南步驟 2（部署）早於步驟 3（多簽產生）。若你已先知道多簽地址，請於部署前填入此項；否則部署腳本會以 `.env` 既有的 `GAS_SPONSOR_ADDRESS` 為預設，**若部署當下尚無該值，`purge_sponsors` 會是空的**，請於步驟 3.2 完成後依該節指示補設。
>
> **dev 與 prod 值不同（重要）**：repo 內提交的 `.env` 之 `SUI_PURGE_SPONSORS` 為 **dev/testnet 多簽地址**（現值 `0x03f1…a46c`），僅供本地開發。**正式上線（mainnet）務必改成步驟 3.1 為 mainnet 產生的 `GAS_SPONSOR_ADDRESS`**；若誤沿用 dev 值，該地址並非 prod sponsor，BFF 將無法在寬限期自動清理，所有問卷都會落入 365 天 fallback。

> [!IMPORTANT]
> 請確保 `SUI_ADMIN_ADDRESS` 帳戶內有充足的 SUI（部署 Mainnet 合約與初始化資金池建議準備至少 **10~20 SUI**）。

### 步驟 2.2：執行合約部署與初始化腳本
於專案根目錄下執行：

```bash
# 安裝相依套件
pnpm install

# 執行部署與初始化
pnpm deploy:Devnet
```
*(註：雖然腳本名稱含 `Devnet`，但其內部讀取 `SUI_NETWORK`。當 `SUI_NETWORK=mainnet` 或 `testnet` 時，會自動向對應網路部署)*。

**該指令將自動完成以下鏈上操作：**
1. 編譯 Move 合約（使用 `--build-env testnet`）。
2. 發行 Package 到 Sui 區塊鏈。
3. 註冊 `SR` (Survey Reward) 與 `SSR` (Stacked Survey Reward) 的 `Currency` 至系統的 `CoinRegistry` (0xc)。
4. 建立 `ProtocolConfig` 並引導初始化 AMM 交易池。
5. 設定 `purge_sponsors`（讀取 `SUI_PURGE_SPONSORS`，未設則以 `GAS_SPONSOR_ADDRESS` 為預設）。終端機會印出 `purge sponsors set: ...`；若印出 `⚠️ … purge_sponsors is empty` 代表兩者皆未設，須於步驟 3.2 後補設。
6. 將鏈上產生的 `SUI_PACKAGE_ID`、`AMM_POOL_ID`、`SR_TREASURY_ID` 等 Object ID **就地寫回**根目錄的 `.env` 檔案中。

**輸出確認：**
部署成功後，終端機會印出 `Deployment complete!`。此時請檢查您的 `.env` 是否已自動附加或更新了以下變數：
- `SUI_PACKAGE_ID`
- `SR_TREASURY_ID`
- `SSR_TREASURY_ID`
- `AMM_POOL_ID`
- `PROTOCOL_CONFIG_ID`
- `SURVEY_REGISTRY_ID`
- `PASS_REGISTRY_ID`
- `ISSUER_CONFIG_ID`
- `VOID_NFT_ID`
- `CLAIM_PASS_SENTINEL_ID`

## 3. 第二階段：多簽代付（Multisig Gas Sponsor）設定

為防止代付熱錢包私鑰洩漏導致代付池資金被盜，SurveySui 使用 **2-of-3 多簽（Multisig）** 帳戶作為代付主體。

### 步驟 3.1：生成多簽金鑰
在專案根目錄執行以下指令：

```bash
pnpm --filter @surveysui/scripts exec tsx src/setup-multisig-sponsor.ts
```

**執行後終端機將印出：**
1. **Sponsor address**：產生的 2-of-3 多簽地址。
2. **K1/K2 私鑰** (`GAS_SPONSOR_PRIV_1` 與 `GAS_SPONSOR_PRIV_2`)。
3. **K3 公鑰** (`GAS_SPONSOR_PUBKEY_3`)。
4. **K3 私鑰** (`GAS_SPONSOR_PRIV_3_OFFLINE`)。

### 步驟 3.2：金鑰與資金分配
1. **備份 K3 私鑰**：請立即將 `GAS_SPONSOR_PRIV_3_OFFLINE` 抄寫或儲存至**離線冷儲存**中，隨後從終端機紀錄中清除。
2. **更新 `.env`**：將產生的 `GAS_SPONSOR_PRIV_1`、`GAS_SPONSOR_PRIV_2`、`GAS_SPONSOR_PUBKEY_3` 與 `GAS_SPONSOR_ADDRESS` 寫入根目錄 `.env`。同時將 `SUI_PURGE_SPONSORS` 設為此 `GAS_SPONSOR_ADDRESS`。
3. **注資多簽地址**：轉入適量 SUI 至產生的 **Sponsor address**（多簽地址）作為代付資金池（勿轉錯至單一私鑰地址）。
4. **補設 purge sponsors（僅當步驟 2 部署早於本步驟時需要）**：若部署當下 `purge_sponsors` 為空（步驟 2.2 印出 `⚠️ … purge_sponsors is empty`），請以 **admin 金鑰**對既有 `ProtocolConfig` 補設一次（毋須重新部署）：

    ```bash
    sui client call \
      --package "$SUI_PACKAGE_ID" \
      --module amm_pool \
      --function set_purge_sponsors \
      --args "$PROTOCOL_CONFIG_ID" '["0x<GAS_SPONSOR_ADDRESS>"]' \
      --gas-budget 10000000
    ```
    > 名單上限 3 名；日後 sponsor 金鑰輪換時，以相同指令**全量覆寫**新名單即可。

## 4. 第三階段：後端 BFF 部署

BFF (`bff/` 目錄) 是常駐 Node.js 服務，負責處理 OAuth 認證、簽發 ticket 以及中轉代付請求。

### 步驟 4.1：準備生產環境環境變數
請在託管平台（如 Fly.io、Render 或 自建 VPS）的環境變數設定後台，注入以下變數：

```env
# ── 基本設定 ──
NODE_ENV=production
PORT=3000
BFF_URL=https://api.yourdomain.com
FRONTEND_URL=https://yourdomain.com

# ── 網路與合約 ID（從步驟 2.2 的根目錄 .env 複製過來） ──
SUI_NETWORK=mainnet
SUI_RPC_URL=https://fullnode.mainnet.sui.io:443
SUI_PACKAGE_ID=0x...
SR_TREASURY_ID=0x...
SSR_TREASURY_ID=0x...
AMM_POOL_ID=0x...
PROTOCOL_CONFIG_ID=0x...
SURVEY_REGISTRY_ID=0x...
PASS_REGISTRY_ID=0x...
ISSUER_CONFIG_ID=0x...
VOID_NFT_ID=0x...
CLAIM_PASS_SENTINEL_ID=0x...

# ── Tier 1：Ticket 簽名與代付多簽 ──
SURVEY_PASS_ISSUER_PRIV=0x...        # BFF 用於簽發 ticket 的私鑰
GAS_SPONSOR_PRIV_1=0x...             # 多簽熱私鑰 1
GAS_SPONSOR_PRIV_2=0x...             # 多簽熱私鑰 2
GAS_SPONSOR_PUBKEY_3=0x...           # 多簽冷公鑰 3
GAS_SPONSOR_MULTISIG_THRESHOLD=2
GAS_SPONSOR_ADDRESS=0x...            # 多簽地址

# ── Tier 2：API 密鑰與身分驗證 ──
SURVEY_PASS_ISSUER_SALT=your_high_entropy_salt_here # 高熵 Salt (必須不等於 default_salt 且長度 >= 12)
ADMIN_SECRET=your_admin_secret_bearer_token       # 用於維護的 Bearer token，可用 openssl rand -hex 32 產生

# ── OAuth 設定（請至 Google/GitHub Developer Console 建立正式專案並獲取） ──
GOOGLE_OAUTH_CLIENT_ID=your-google-client-id
GOOGLE_OAUTH_CLIENT_SECRET=your-google-client-secret
GITHUB_OAUTH_CLIENT_ID=your-github-client-id
GITHUB_OAUTH_CLIENT_SECRET=your-github-client-secret

# ── World ID 驗證（正式 Production 環境憑證） ──
WORLDCOIN_APP_ID=app_prod_...
WORLDCOIN_RP_ID=yourdomain.com
WORLDCOIN_SIGNING_KEY=your-worldcoin-signing-key
WORLDCOIN_ACTION=your-action-name

# ── Email OTP 寄送 ──
RESEND_API_KEY=re_...
EMAIL_FROM=noreply@yourdomain.com

# ── Gas 模式設定 ──
GAS_STATION_MODE=do                  # 使用 Cloudflare Worker 做分散式代付
GAS_STATION_URL=https://gas-station.yoursubdomain.workers.dev
GAS_STATION_SHARED_SECRET=your-hmac-shared-secret # 用於 BFF 與 Worker 通訊的 HMAC Secret
```

> [!WARNING]
> 部署 BFF 時，**絕對不能**設定 `SUI_ADMIN_PRIVATE_KEY`。BFF 在初始化時若檢測到該變數會崩潰以保障安全。

### 步驟 4.2：構建與啟動 BFF
BFF 通常使用 Docker 或 Node 運行：

```bash
cd bff
# 編譯打包
pnpm build
# 啟動生產環境服務
pnpm start
```

## 5. 第四階段：Cloudflare Gas Station Worker 部署

Gas Station Worker (`workers/gas-station/` 目錄) 負責接收 BFF 的代付請求，使用 SQLite D1 做防刷頻控，並對交易進行多簽簽章。

### 步驟 5.1：建立並配置 D1 資料庫
在 Cloudflare CLI (`wrangler`) 中建立 D1 資料庫：

```bash
cd workers/gas-station
npx wrangler d1 create surveysui-db-prod
```
系統會輸出資料庫的 `database_id` 與 `database_name`。

請開啟 `workers/gas-station/wrangler.toml`，在對應區段填入 D1 綁定與環境變數：

```toml
name = "gas-station"
main = "src/index.ts"
compatibility_date = "2024-01-01"

[vars]
SUI_RPC_URL = "https://fullnode.mainnet.sui.io:443"
SUI_NETWORK = "mainnet"
GAS_STATION_MODE = "do"
GAS_SPONSOR_MULTISIG_THRESHOLD = "2"
# 填入步驟 2.2 生成的合約 ID
SUI_PACKAGE_ID = "0x..."

[[d1_databases]]
binding = "DB"
database_name = "surveysui-db-prod"
database_id = "您的-d1-database-id"

[[durable_objects.bindings]]
name = "GAS_STATION_DO"
class_name = "GasStationDO"

[[migrations]]
tag = "v1"
new_classes = ["GasStationDO"]
```

### 步驟 5.2：初始化資料庫 Schema
使用 wrangler 將 schema 部署到 Cloudflare D1 生產環境：

```bash
# 執行 schema migration
npx wrangler d1 migrations apply DB --remote
```

### 步驟 5.3：注入 Worker 敏感 Secrets
使用 `wrangler secret put` 將 Tier 1 與驗證密鑰寫入 Cloudflare：

```bash
npx wrangler secret put GAS_SPONSOR_PRIV_1
# (貼入 K1 私鑰)

npx wrangler secret put GAS_SPONSOR_PRIV_2
# (貼入 K2 私鑰)

npx wrangler secret put GAS_SPONSOR_PUBKEY_3
# (貼入 K3 公鑰)

npx wrangler secret put GAS_STATION_SHARED_SECRET
# (貼入與 BFF 相同的 HMAC 密鑰)

npx wrangler secret put SURVEY_PASS_ISSUER_PRIV
# (貼入 ticket 簽發私鑰，供 DO 驗證 ticket 使用)
```

### 步驟 5.4：部署 Worker
```bash
npx wrangler deploy
```

## 6. 第五階段：前端 Cloudflare Pages 部署

前端為 Vite 靜態網站。

### 步驟 6.1：設定環境變數
前端打包時需要讀取根目錄 `.env` 的 `VITE_*` 公開變數（這些變數皆為非敏感內容）。請確認 `.env` 內包含以下設定：

```env
# ── 前端與後端 API 對接 ──
VITE_BFF_URL=https://api.yourdomain.com
VITE_NETWORK=mainnet

# ── 鏈上合約 ID 映射 ──
VITE_PACKAGE_ID=0x...         # 同 SUI_PACKAGE_ID
VITE_SR_TREASURY_ID=0x...     # 同 SR_TREASURY_ID
VITE_SSR_TREASURY_ID=0x...    # 同 SSR_TREASURY_ID
VITE_AMM_POOL_ID=0x...        # 同 AMM_POOL_ID
VITE_SURVEY_REGISTRY_ID=0x... # 同 SURVEY_REGISTRY_ID
VITE_PASS_REGISTRY_ID=0x...   # 同 PASS_REGISTRY_ID
VITE_ISSUER_CONFIG_ID=0x...   # 同 ISSUER_CONFIG_ID
VITE_ADMIN_ADDRESS=0x...      # 同 SUI_ADMIN_ADDRESS

# ── 其他前端設定 ──
VITE_PURGE_GRACE_MS=86400000  # 答卷銷毀寬限期 (例如 24 小時)
```

### 步驟 6.2：建置與發布
您可以使用 Cloudflare Pages 綁定 GitHub Repository 進行自動 CI/CD，或是直接手動編譯部署：

```bash
# 切換至前端目錄
cd frontend

# 編譯打包 (Vite 會將 VITE_* 寫入 build bundle 中)
pnpm build

# 使用 wrangler 部署至 CF Pages
npx wrangler pages deploy dist --project-name surveysui-frontend
```

## 7. 上線後驗證與維護 SOP

部署完成後，請依序完成以下檢查：

### 7.1 健康檢查與基礎連線
- 呼叫 BFF 的健康檢查 API：`GET https://api.yourdomain.com/health`，確認回傳 `status: "ok"` 且無金鑰缺失警告。
- 檢查 BFF 日誌中是否含有 `assertSecureEnv` 拋出的錯誤，確保 Admin Key 已安全清除。

### 7.2 代付與合約互動驗證
- 打開前端網站，點擊「連接錢包」。
- 嘗試發起一個 mint SurveyPass 的流程，確認：
  1. BFF 成功簽發 pass ticket。
  2. 瀏覽器將交易傳送給 BFF。
  3. BFF 將代付請求中轉給 Cloudflare Worker `gas-station`。
  4. Worker 成功使用多簽 (K1+K2) 簽名並廣播交易。
  5. 交易上鏈成功，且您的錢包**沒有支付任何 SUI**（確認代付生效）。

### 7.3 日常維護：代付池注資
- 請定期監控多簽代付帳戶（`GAS_SPONSOR_ADDRESS`）的 SUI 餘額。
- 當餘額不足時，直接將 SUI 轉入該地址即可，無須重啟服務。

### 7.4 緊急應變（私鑰洩漏）
- 若發生 **Issuer 私鑰** 或 **Sponsor 熱金鑰** 洩漏，請立即參考 [安全指引.md](./安全指引.md) 的 **§9 洩漏應變流程** 進行合約變更與金鑰輪換。

### 7.5 問卷生命週期與 purge 權限
- **建立期上限**：合約於建立時強制問卷有效期（deadline）不得超過建立當下 **92 天**（前端另有 3 曆月驗證）。超過則交易以 `EDeadlineTooFar` 中止。
- **銷毀寬限期**：問卷關閉 / 到期後預設 **92 天**（`DEFAULT_PURGE_GRACE_MS`，前端可由 `VITE_PURGE_GRACE_MS` 覆寫寫入 vault）寬限期屆滿，授權者方可 `purge` 永久銷毀題目與答卷。
- **誰能銷毀**：問卷發起者（關閉後立即）、`purge_sponsors` 名單 / 合約 admin（寬限期後，BFF 自動清理）、其他任何人（僅 liveness 備援，需到期 + **365 天**）。
- **sponsor 輪換**：若 BFF 多簽 `GAS_SPONSOR_ADDRESS` 變更，務必由 admin 重新執行步驟 3.2 的 `set_purge_sponsors`（全量覆寫，上限 3 名），否則新 sponsor 的自動 purge 會落入 365 天備援路徑。
