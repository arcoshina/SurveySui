# SurveySui V2 — 改版目標 & 設計

## Context

V1 已完成（commit `23a68b5`），SurveySui MVP 三層架構（Frontend / BFF / Move Contracts）跑通 Flow A→B→C，整體資金流可運作。

V2 的目的：

1. **修復 V1 上線後暴露的 bug 與安全缺口**（sSSR 畸零、Markdown 預覽、合約端內容驗證、重複檢查等）。
2. **重新整理金流結構**：把 V1 的 vault 0.3% 手續費換成兩段式設計——**有效費率 = 總費率 20% × 折扣率（預設 50%）= 10%**；同時把發起問卷 PTB 改成七步驟原子交易，**新鑄 sSSR 不再傳給發起者**。
3. **規畫 SurveyPass 身分認證簽發**（email / KYC / zkLogin），不一定全部 V2 實作，但設計要拍板。
4. **規畫匿名投票方案**（下一版實作；本版只出設計）。

本檔的角色：定義 V2 的**改版範圍與設計決策**——在「**整體要做什麼**」（[專案目標.md](專案目標.md)）與「**怎麼測 / 怎麼證明做對了**」（[V2_TDD.md](V2_TDD.md)）之間，記錄**這次改了什麼、為什麼這樣改**。**V1 已固化的設計決策、模組職責、測試框架不再重述**，請以 [History/V1_TDD.md](History/V1_TDD.md) 為基底，本檔僅記錄差分。

> **進度追蹤請見 [V2_Tasks.md](V2_Tasks.md)**。

## 新／改 設計決策

> 與 [V1_TDD §已對齊的設計決策](History/V1_TDD.md) 並列；衝突時以本表為準。

| 議題 | V1 決策 | V2 決策 | 理由 |
| --- | --- | --- | --- |
| AMM 初始比率 | 1 MIST → 1000 sSSR base（已寫死於 `amm_pool::INITIAL_SSSR_PER_SUI`） | **保留**，V2 任務「確認 1 SUI = 1000 SSR」屬於文檔對齊，不改合約常數 | 已被 V1 驗證，無理由改 |
| 平台手續費結構 | `survey_vault` 入金時收 30 bps，admin 收 sSSR | **有效費率 = 總費率 20% × 折扣率（預設 50%）= 10%**（公式 `total × discount`，非 `total - discount`）；費率分拆在 PTB 內由獨立 step 處理 | 抽成模型要有「正常費率」與「行銷期折扣」兩種狀態；V1 30 bps 沒有調整空間 |
| 費率參數的管理面 | 寫死在常數 | 寫進共享物件（`amm_pool::Pool` or 獨立 `FeeConfig`），admin-only update；**不做後台介面**，直接前端/合約硬編 | V2_Tasks 明確寫「直接更新前端和合約，不需要 admin 介面」，省 sprint |
| 發起問卷 PTB 結構 | 4 步：`invest_and_mint`（mint sSSR 給發起者）→ `vault::create`（從發起者扣 sSSR + 收 30 bps 手續費）→ `registry::register` → 完成 | **7 步原子交易**：① 預估 → ② 建立 Vault（空殼）→ ③ 發起者既有 sSSR 注入 Vault → ④ AMM 注資（mint sSSR 入 vault，不給發起者）→ ⑤ 合併 Vault 中的 sSSR → ⑥ 費率分拆（vault → admin treasury）→ ⑦ 註冊問卷 | 移除「sSSR 在發起者錢包停留」的中間態，杜絕中介竄改；既有 sSSR 先入庫再注資，方便鏈上對 vault 餘額做 invariant 檢查；費率分拆獨立 step 方便未來換折扣比率 |
| 發起者既有 sSSR 折抵 | 不支援 | **支援**：若發起者已持有 sSSR，PTB 步驟 ③ 先把既有 sSSR 注入 vault，步驟 ④ AMM 注資數量 = 缺口；步驟 ⑤ 合併成單一 balance | 配合「不傳給發起者」原則，且 V1 demo 已顯示重複發問卷需要折抵 |
| 問卷內容驗證 | 全前端 parse + 驗證 | **合約端再驗一次完整結構**（題型白名單、選項數量上界、總長度上界、防空題、防重複題 ID）；前端仍負責 UX 驗證 | V2_Tasks P1：前端內容可被竄改 |
| 問卷重複檢查 | 無 | **合約端對 `registry::register` 做內容雜湊去重 + 完整結構驗證**；先查重複，再驗完整性 | 防 spam 與內容污染 |
| 答案上鏈格式 | 加密整份問卷答案（含題目對應） | **只記錄結果（依題目順序對應）+ 加密**；考慮加冗餘欄位（如 question_count / schema_hash）防錯位 | 降 gas、降資料量；題目模板已存在 `survey_registry` |
| SurveyPass 簽發 | admin 手動 `issue` | **整體待規劃，實作時再評估**。目標：**驗證「不重複的真人」**（單一 email 不夠，要結合其他訊號）。候選路徑：(a) email OTP、(b) 類 KYC、(c) Sui zkLogin、(d) Google OAuth；其中 zkLogin / Google OAuth 的機制要**先確定**（兩者是登入入口還是驗證來源、與連自有錢包是否相容） | V1 admin 手動發 pass 無法防女巫；本表先列待規劃事項，不在 V2_TDD 拍板 |
| SurveyPass 與連錢包流程 | 一個入口 | **待規劃**：驗證流程**可能會要求連錢包**（例如要把驗證結果綁定地址、或直接以錢包簽 challenge）。是否要與「填答前的連錢包」整合到同一個 flow，需在實作時連同 (c)(d) 的機制決定 | 解耦或整合各有利弊，目前資訊不足以拍板 |
| 匿名投票 | 不在 MVP | **下一版實作**；本版只交設計：使用 zk proof / nullifier 結構，但與 SurveyPass 防女巫的相容性需處理 | V2_Tasks P0：「下一版要實作」 |
| 加密公鑰存放 | 每次填答即時簽名 → 衍生對稱金鑰 → 上鏈無公鑰 | **兩步驟簽名發起流程**：① 按「設定加密金鑰」→ `signPersonalMessage` → 衍生 X25519 keypair 存於 React state；② 按「發布問卷」→ `signAndExecuteTransaction`。Keypair 不寫任何持久化 storage，component unmount 即清除。 | 兩次簽名在技術上不可避免（公鑰必須在 PTB 建構前確定）；拆成兩個獨立按鈕讓每次 wallet 彈窗都有明確用途說明，UX 比「一鍵連彈兩次」更可接受。 |
| Gas Station 失效處理 | 失敗即報錯 | **fallback：BFF 不可用時，前端直接做 dry-run（client RPC），通過後改走自付 gas 流程**並明確告知 UX 切換 | V1 後端失效會完全卡住，影響 demo |

## 發起問卷 PTB 七步驟（V2 核心）

> 取代 [V1_TDD §資金流](History/V1_TDD.md) 中「Flow A 三步」的版本。整個流程仍是**一筆 PTB 原子交易**，但拆成 7 個 step 方便驗證與費率分拆。

```
發起者按下「預覽問卷 + 注資」按鈕
   │
   │  ┌──────────────────────────────────────────────┐
   │  │  ① 預估（純前端，PTB 之前）                   │
   │  │     - 讀 Pool.total_sui_invested              │
   │  │     - 計算 net_sssr / 既有 sSSR 折抵 / 缺口    │
   │  │     - 計算 SUI 注資量 + effective_fee_bps     │
   │  └──────────────────────────────────────────────┘
   │
   ▼  以下打包成一筆 PTB
   ② survey_vault::create_empty(reward_per_response, max, conditions, content_hash)
        → 產生空殼 Vault<sSSR>（shared object）

   ③ survey_vault::deposit_existing_sssr(vault_ref, sssr_coin_from_creator)
        → 發起者既有 sSSR 先注入 vault（無折抵時帶 zero coin，仍走此 step 保持流程一致）

   ④ amm_pool::invest_and_mint(sui_in, vault_ref)
        → 池中 SUI ↑、mint 新 sSSR **直接灌入 vault**（不傳給發起者）

   ⑤ survey_vault::merge_balances(vault_ref)
        → 把 vault 內既有 sSSR balance 與新鑄 sSSR balance 合併為單一 balance；
          鏈上 invariant 檢查：合併後總額 ≥ net_sssr（= per_response × max）

   ⑥ survey_vault::split_fee_to_treasury(vault_ref, FeeConfig)
        → 依 effective_fee_bps 把對應 sSSR 從 vault 提撥到 admin treasury

   ⑦ survey_registry::register(vault_id, content_hash, encrypted_blob, schema_hash)
        → 鏈上去重 + 結構驗證，發送 SurveyRegistered 事件
```

**為什麼這樣拆**

- **②→③→④**：先建空 vault，再讓既有 sSSR 入庫，最後 AMM 注資補缺口。把「新鑄 sSSR 的 owner」直接設為 vault，發起者錢包**完全不會出現新鑄 sSSR**。
- **③ 先於 ④**：既有 sSSR 先進 vault，⑤ 合併時可以對 vault 餘額做 invariant 檢查（總額 ≥ 目標），把鏈下估算誤差擋在合約層。
- **⑤ 獨立**：合併與費率分拆解耦；合併本身可加 invariant，未來若改 AMM 公式只動 ④。
- **⑥ 獨立**：未來改折扣比率只動這個 step 與 `FeeConfig`，不會牽動 vault `create` 邏輯。
- **⑦ 最後**：所有資金狀態都進入 vault 後才 register，事件可以一次帶齊最終狀態。

**Failure mode**

- 任一 step abort → 整筆 PTB rollback，發起者錢包 / vault / pool / treasury 全部回滾。
- ⑤ 合併後總額 < net_sssr → abort `EInsufficientVaultBalance`；通常代表前端估算 SUI 量有誤（bonding curve 在 step 之間被別人推動了）。
- ⑦ 內容雜湊重複 → abort `EDuplicateSurvey`；UI 提示「您似乎已發過相同問卷」。
- ⑦ 結構不合法 → abort `EInvalidSchema`；UI 對應顯示題目錯誤位置（前端事前已驗，理論上不該打到這裡，但留 defensive check）。

## SurveyPass 認證簽發（定稿版，S5.1 拍板完成）

> 完整設計架構見 [docs/專案 KYC方案.md](專案%20KYC方案.md)。  
> 測試規格見 [V2_TDD.md §S6](V2_TDD.md)。

### V2 MVP 拍板決策

| 問題 | 決策 |
|------|------|
| zkLogin / Google OAuth 定位 | **MVP 不使用**。驗證源為 Email OTP（`SRC_EMAIL`）；Social OAuth / zkLogin 留 V3 評估 |
| 驗證是否要求連錢包 | **是**。IssuanceTicket 綁定 `owner: address`，驗證前必須先連錢包 |
| 真人驗證訊號 | **Email OTP only（MVP）**；nullifier = `hash("email" \|\| email_address)`；一個 email 只能對應一張有效 Pass（INV-8）|
| BFF 角色 | **ticket-only**（INV-7）。BFF 負責：① 發 OTP、② 驗 OTP、③ 算 nullifier_hash、④ 簽 Ed25519 IssuanceTicket、⑤ 回傳 ticket JSON。不送 TX，不持 admin key |

### 信任層級（定稿）

Email OTP → tier 0（輔助訊號）；Social → tier 1；Self Protocol / World ID → tier 2（最高）。  
V2 MVP 只實作 Email OTP（tier 0）；tier 1–2 留待後續版本。

### 長期相容性預留

- `SurveyPass.vote_commitment` 欄位預留（S5.2 匿名投票用），不破壞 MVP 結構
- `CredentialSlot` 採 dynamic_field 掛載，未來新增 Social / World ID source 無需改 struct

## 匿名投票方案（設計交付完成，S5.2 完成）

> 完整設計見 [docs/V2_AnonymousVoting_Sketch.md](V2_AnonymousVoting_Sketch.md)。  
> 以下為摘要，詳細內容請見設計文件。

### 設計方向（定稿）

- 採用 **Semaphore-style Groth16 ZKP**（與 World ID 底層相同），利用 Sui 原生 `sui::groth16` 模組在鏈上驗證
- **雙 nullifier 模型**：issuance nullifier（防女巫）與 vote nullifier（防重複投票）獨立運作，不互相洩漏
- **Vote secret 衍生**：`s = HMAC(wallet_privkey, "surveysui-vote-v1")`，無需額外存儲，任何時候可重新衍生
- **Relayer 匿名**：Gas Station 作為 TX sender，隱藏投票者鏈上 address
- **Re-vote 機制**（V4 目標）：投票期間可覆蓋舊票，達成抗脅迫性

### 與 SurveyPass 的衝突解法

SurveyPass 新增 `vote_commitment: Option<vector<u8>>` 欄位；Pass 持有者可選擇性呼叫 `register_vote_commitment` 加入 CommitmentTree，實現匿名集合。現有 SurveyPass 功能不受影響。

### 國際標準合規對照

V3 達成：匿名性、唯一性、個人/全局可驗性；V4 補強：抗脅迫性（Re-vote）。無收據性為 V5+ 研究議題。
