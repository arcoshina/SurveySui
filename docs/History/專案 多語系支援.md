# SurveySui 多語系支援計畫

本文件記錄專案多語系支援（i18n）架構盤點，並針對新增**日文（JA）**、**韓文（KO）**、**西班牙文（ES）**的支援提供具體的實作與切換邏輯計畫。

---

## 一、 目前 i18n 運作邏輯分析

目前專案已實作了一套自定義、輕量且**型別安全（Type-Safe）**的多國語言架構，主要程式碼集中於前端 `frontend/src`：

### 1. 字典管理 (`frontend/src/i18n`)
- **語系架構**：目前僅支援 `ZH`（繁體中文）與 `EN`（英文）。
- **模組化 Namespace**：翻譯字典依照功能模組拆分為 9 個檔案，例如 `navbar.ts`、`survey.ts` 等。
- **型別正本設計**：
  - `zh/` 資料夾下的翻譯做為**型別正本**，定義並導出該 Namespace 的字典型別（例如 `export type NavbarDict = typeof navbar`）。
  - `en/` 及其他未來語系會引入該型別定義（例如 `const navbar: NavbarDict = { ... }`），以確保翻譯鍵的完整性與一致性。若有遺漏，編譯階段即會報錯。
- **動態插值（Interpolation）支援**：
  - 許多詞條被設計為接收參數並回傳字串的函數（例如 `gasSponsorQuota: (remaining, max) => string`），能以 `t.gasSponsorQuota(remaining, max)` 的方式進行型別安全的動態插值。
- **集中註冊與 Hook (`frontend/src/i18n/index.ts`)**：
  - 透過 `dictionaries` 集中註冊各 Namespace。
  - 使用 `useT(ns)` Hook，根據 `LanguageContext` 目前的語系，返回對應的翻譯物件。

### 2. 語系狀態管理 (`LanguageContext.tsx`)
- **支援型別**：`export type Language = 'ZH' | 'EN'`
- **狀態儲存**：寫入 `localStorage`（鍵名為 `surveysui:lang`）。
- **URL 同步**：
  - 語系與網址 `?lang=zh|en` 雙向同步。
  - 當網址參數變動時套用該語系；當網址無參數時，自動補上當前語系，使網址持續維持 locale 縮寫。
  - *注意：更新網址時保留了 `location.hash`（因為問卷加密金鑰是放在 hash 中，若丟失將導致無法解密問卷）。*
- **初始化偵測**：優先序為 `網址 ?lang` ➡ `localStorage` ➡ `瀏覽器系統語言 (navigator.language)` ➡ `預設 ZH`。
- **HTML 同步**：套用時更新 `<html lang="zh-TW|en">`。
- **切換機制**：目前採用 `toggleLang` 二元循環切換。

## 二、 語系切換與 Fallback 設計規格

根據需求，未來多語系擴充後的切換與初始化邏輯設計規格如下：

1. **偵測系統語言 (System Language Detection)**：
   - 偵測訪客的瀏覽器系統語言。
   - 如果系統語言在**支援的本地語系範圍內**（中文 `ZH`、日文 `JA`、韓文 `KO`、西班牙文 `ES`），則首頁/介面自動以該語系呈現。
   - 如果訪客的系統語言**不在支援範圍內**，則一律 **fallback 顯示英文 (`EN`)**。
2. **切換機制 - 切換本地或英文 (Toggle Local or English)**：
   - 若訪客的系統語言是支援的本地語言之一（如日文 `JA`）：
     - 點擊語言切換按鈕時，會在「該本地語言（`JA`）」與「英文（`EN`）」之間進行互切。
   - 若訪客的系統語言是英文（`EN`），或是其他不被支援的語系（以英文 fallback 呈現）：
     - 此時**點擊切換亦可正常執行與渲染，只是它是從「本地語言（英文）」切換為「國際語言（依然是英文）」，實際上看不出切換效果**。
3. **網址手動切換優先**：
   - 訪客可手動在網址列帶上支援語系（例如 `?lang=ja`、`?lang=ko`）以強行切換語系。
   - 若網址列帶入的 `lang` 參數是不支援的語系（例如 `?lang=fr`），則**一律顯示英文 (`EN`)**。

## 三、 擴充日文、韓文、西班牙文之具體修改方案

### 1. 擴充與重構 `LanguageContext.tsx` 狀態管理
- **語言型別定義**：
  ```typescript
  export type Language = 'ZH' | 'EN' | 'JA' | 'KO' | 'ES'
  ```
- **記錄訪客的系統/本地語言狀態**：
  在 `LanguageProvider` 中，除了當前的 `lang`，還需要保存偵測出來的「本地語言（`detectedLocalLang`）」狀態：
  ```typescript
  // 記錄偵測到的本地語言，如果偵測到是英文或不支援的語言，則為 'EN'
  const [localLang, setLocalLang] = useState<Language>('EN')
  ```
- **系統語言偵測與本地語言綁定 (`detectSystemLang`)**：
  ```typescript
  /** 偵測系統語言，若支援則回傳，否則回傳 'EN' */
  function getSupportedSystemLang(): Language {
    if (typeof navigator === 'undefined') return 'EN'
    const list = navigator.languages?.length ? navigator.languages : [navigator.language]
    for (const l of list) {
      if (!l) continue
      const code = l.toLowerCase()
      if (code.startsWith('zh')) return 'ZH'
      if (code.startsWith('ja')) return 'JA'
      if (code.startsWith('ko')) return 'KO'
      if (code.startsWith('es')) return 'ES'
      if (code.startsWith('en')) return 'EN'
    }
    return 'EN' // 預設
  }
  ```
  在初始化時：
  - 偵測系統語言 `sysLang = getSupportedSystemLang()`。
  - 將 `localLang` 設為 `sysLang`。
  - 初始化語言優先序：網址 `?lang` (在支援範圍內) ➡ `localStorage` ➡ 系統語言 (在支援範圍內) ➡ 後備 `EN`。
- **語系對照輔助函數更新**：
  原有的對照函數必須擴展以支援五種語系：
  ```typescript
  const toUrlCode = (l: Language) => {
    switch (l) {
      case 'ZH': return 'zh'
      case 'JA': return 'ja'
      case 'KO': return 'ko'
      case 'ES': return 'es'
      default: return 'en'
    }
  }
  const toHtmlLang = (l: Language) => {
    switch (l) {
      case 'ZH': return 'zh-TW'
      case 'JA': return 'ja'
      case 'KO': return 'ko'
      case 'ES': return 'es'
      default: return 'en'
    }
  }
  ```
- **輸入正規化 (`normalizeLang`)**：
  必須嚴格限制僅回傳支援的五種語系，其餘一律回傳 `null` (進而 fallback 到英文)。
  ```typescript
  function normalizeLang(raw: string | null | undefined): Language | null {
    if (!raw) return null
    const v = raw.trim().toLowerCase()
    if (v.startsWith('zh')) return 'ZH'
    if (v.startsWith('en')) return 'EN'
    if (v.startsWith('ja')) return 'JA'
    if (v.startsWith('ko')) return 'KO'
    if (v.startsWith('es')) return 'ES'
    return null // 不支援的語言 code 統一由呼叫端 fallback 處理
  }
  ```
- **優化後的 `toggleLang` 機制與 `nextLang` 導出**：
  為了避免英文系統或不支援系統語言的使用者在點擊語言切換時「沒有任何可見變化」而被誤認按鈕失效，當 `localLang === 'EN'` 時，我們將二元切換的目標本地語言預設為 `'ZH'`（繁體中文）。
  此外，在 `LanguageContext` 的 Props 中，我們需要額外導出 `nextLang` 來表示點擊切換後下一個會套用的語言，使 UI 能夠進行動態切換提示：
  ```typescript
  // 計算下一個語言
  const targetLocal = localLang === 'EN' ? 'ZH' : localLang
  const nextLang = lang === targetLocal ? 'EN' : targetLocal

  const toggleLang = () => applyLang(nextLang)
  ```

### 2. UI 按鈕與動態提示 (`Navbar.tsx`)
- **按鈕渲染與互動**：
  - 不需要隱藏按鈕，Navbar 上的語言切換按鈕照常渲染且可被正常點選。
  - 當點擊按鈕時，調用 `toggleLang()` 將語系切換至 `nextLang`。
- **動態切換提示 (Tooltip & Aria-label)**：
  - 由於支援的語系變多且二元切換的目標是動態的，按鈕的 hover 提示/title 不能再使用寫死的靜態字串。
  - 應在各語系的 `navbar.ts` 字典中定義切換至各語言的翻譯鍵：
    - 中文 `ZH` 增加 `switchLangToEn`, `switchLangToZh`, `switchLangToJa`, `switchLangToKo`, `switchLangToEs` 等鍵。
    - 其他語系仿照辦理，並使用 `NavbarDict` 保持型別安全。
  - 在 `Navbar.tsx` 中，依據從 `LanguageContext` 取得的 `nextLang`，動態顯示對應的切換提示文字（例如：目前 `nextLang === 'JA'`，則提示為 `switchLangToJa` 的文案）。

### 3. 建立並註冊翻譯字典
1. **建立新目錄與實作字典**：
   - 建立 `frontend/src/i18n/ja/`、`frontend/src/i18n/ko/`、`frontend/src/i18n/es/`。
   - 在各目錄下建立 9 個對應的翻譯檔，必須 import `zh` 下的型別定義（如 `NavbarDict`、`SurveyDict` 等）以保證編譯期型別安全。
2. **在 `frontend/src/i18n/index.ts` 中註冊**：
   ```typescript
   export const dictionaries = {
     navbar: { ZH: zhNavbar, EN: enNavbar, JA: jaNavbar, KO: koNavbar, ES: esNavbar },
     // (其餘 8 個 namespace 同步註冊...)
   } as const
   ```

### 4. BFF 驗證信 (Email OTP) 多語系支援
- **前端 `AuthPage.tsx` 傳送語系代碼**：
  - 當使用者點擊「發送驗證碼」時，前端在發送 POST 請求至 `/auth/email/otp` 時，除了 `email` 之外，必須將當前語系 `lang` 帶在請求 Body 中：
    ```typescript
    body: JSON.stringify({ email, lang })
    ```
- **BFF 端接收與發送多語系郵件**：
  - 在 `bff/src/auth/handler.ts` 的 `/auth/email/otp` 路由中，自請求的 Body 接收 `lang?: string`。
  - 呼叫 `sendOtpEmail(email, code, lang)`。
  - 在 `bff/src/email/sender.ts` 中擴充 `sendOtpEmail` 函數，定義中文、英文、日文、韓文與西班牙文的主旨與 HTML 內容，並依據 `lang` 進行分支發送（若傳入不支援的語系代碼，則預設 fallback 至英文 `EN` 範本）。
