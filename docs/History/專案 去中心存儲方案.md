# 專案 去中心化儲存方案

## 專案目標
將加密後的問卷內容 (Survey Content) 與答卷內容 (Survey Answers) 由「Sui 鏈上直接儲存」轉換為「去中心化儲存 + 鏈上 Hash 指針」的混合方案。
* **核心價值**：
  1. 鏈上交易參數的大小與資料內容長度解耦，使每次填答的 Gas 消耗維持在極低且高度恆定的狀態。
  2. 項目方在問卷上架注資時，不需要進行複雜的合併交易 Dry-run 模擬，可以用簡單的公式進行 100% 精準的 Gas 費用預估。
  3. 保證 Web3 的抗審查與無單點故障 (No Single Point of Failure) 精神，即使後端 BFF 離線，使用者仍能正常註冊與填答。

---

## 採用方案：大小分流的「動態混合儲存方案 (Dynamic Hybrid Storage Mode)」

為了同時兼顧「小檔案的極速體驗與低複雜度」以及「大檔案的 Gas 節省」，本系統不採用單一的分離儲存，而是採用**「動態混合儲存方案」**。前端會根據加密加密後的資料體積，自動在 **「直接上鏈模式」** 與 **「去中心化分離模式」** 之間進行切換。

本系統全面採用 **「先上傳存儲，確認成功後再注資上鏈」** 的序列化流程，100% 避免死指針造成的資料丟失風險。此外，**項目方不代墊任何儲存費用**，去中心化儲存費用完全由發起者自行支付（題目卷）或預先提撥（答卷）。

### 1. 動態分流門檻與模式

| 儲存對象       | 臨界門檻值 | 模式分配運作                                                                                                                                                           |
| :------------- | :--------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **問卷題目卷** | **10 KB**  | $\le 10\text{ KB}$：**直接上鏈模式**（免直傳上傳，直接存入 Move 物件）<br/>$> 10\text{ KB}$：**去中心化分離模式**（直傳 Walrus/IPFS，鏈上僅記指針與 Hash）             |
| **答卷內容**   | **4 KB**   | $\le 4\text{ KB}$：**直接上鏈模式**（將加密答卷直接作為交易參數寫入合約）<br/>$> 4\text{ KB}$：**去中心化分離模式**（直傳 Walrus/IPFS，鏈上僅記 ID 與 Nullifier 綁定） |

---

### 2. 運作流程與架構

#### A. 發布問卷（前端自動分流）
1. **第一步：本地加密 (Encrypt)**
   * 前端本地打包問卷 JSON，並在本地加密。
   * 前端檢測加密後的密文大小。
2. **第二步：動態上傳 (Upload) 或跳過**
   * **若大小 $\le 10\text{ KB}$ (直接上鏈模式)**：
     * 前端跳過此步驟，UI 顯示：`「⚡ 小於 10KB，優化為直接上鏈模式，跳過第三方儲存上傳」`。
   * **若大小 $> 10\text{ KB}$ (去中心化分離模式)**：
     * 前端將加密檔案上傳至去中心化儲存網路（優先使用 Walrus，IPFS 為備用）。
     * **UI 透明標註**：畫面上明確標註目前的去中心化儲存服務（例如："*正在將問卷上傳至 Walrus 去中心化網路...*" 或 "*正在將問卷儲存於 IPFS 節點...*"），讓發佈者明確掌握檔案託管位置。上傳完成後取得 `survey_blob_id` 或 CID。
3. **第三步：注資與發布 (Fund & Register)**
   * 前端喚起使用者錢包簽署**一次**交易。
   * **直接上鏈模式**：交易直接將加密問卷數據寫入合約，並存入受訪者獎勵池。
   * **去中心化分離模式**：交易寫入 `survey_blob_id` 與防篡改的 `survey_content_hash`，並注入受訪者獎勵池，以及預先提撥的答卷去中心化儲存預付款（用於受訪者大於 4KB 答卷時的扣款）。

#### B. 填答答卷（前端自動分流）
1. **本地加密與 Padding**
   * 填答者在前端本地完成資料加密。
   * **簡答題上限填充 (Padding)**：問卷發起人可在建立問卷時設定簡答題字數上限（預設 100 字）。前端會根據上限對該簡答題進行長度填充再加密，防範透過檔案大小猜測答案。
   * **隨機 Salt**：在加密前，前端必須對答卷內容加入隨機產生的 Salt，確保相同回答也會產生獨一無二的 `answer_blob_id`，防止合約去重衝突。
2. **提交與上鏈**
   * **若大小 $\le 4\text{ KB}$ (直接上鏈模式)**：
     * 前端直接喚起填答者錢包，將加密答卷密文作為參數呼叫 `claim` 合約。受訪者僅需點擊一次，體驗流暢。
   * **若大小 $> 4\text{ KB}$ (去中心化分離模式)**：
     * 前端在背景將密文直傳至 Walrus/IPFS。上傳成功取得 `answer_blob_id` 後，前端自動喚起錢包，將 `answer_blob_id` 上鏈。儲存費用由發起者預先提撥的資金池中扣除。

#### C. 身分認證 Nullifier 與 ID 綁定
* **防冒領機制**：當走去中心化分離模式時，在產生 Nullifier 證明或交易簽名時，必須將 `answer_blob_id` 作為 Public Input 或簽名的一部分。合約校驗確保「此身分（Nullifier）只能提交此特定的答卷指針」，防止 ID 被他人攔截套用。

#### D. 後端快取與 Pin 增強（零信任校驗）
* **前後端零信任**：前端不持有任何項目方的 Pinning API Key。BFF 後端不直接信任前端上鏈的任何指針。
* **問卷（題目卷）防篡改校驗**：BFF 監聽到問卷註冊事件後，若為去中心化分離模式，會下載該檔案並在後端本地計算 Hash，比對是否與鏈上發起者註冊的 `survey_content_hash` 100% 一致。核對無誤且檔案小於安全上限（如 10MB）時，後端才執行 Pin 請求與 R2/S3 快取。
* **答卷大小校驗**：答卷有精確的大小預期。BFF 檢測到走去中心化分離模式的答卷檔案大小超出合理大小（如 > 50KB）則拒絕 Pin 與快取。若為直接上鏈模式，則直接對鏈上事件發射的密文進行快取。

## 前端 UI 與使用者體驗設計

為了提升 Web3 應用的質感（Premium Aesthetic）與互動信賴感，前端 UI 針對動態儲存模式進行專屬設計：

### 1. 透明的狀態指示器 (Mode & Storage Badges)
* **模式標籤**：前端發布或填答介面會配備一個精美的 Badge：
  * 直接上鏈模式顯示：`⚡ 直傳上鏈模式 (4.2KB / 門檻 10KB)` (綠色/青色漸變)
  * 去中心化儲存模式顯示：`🌐 Walrus 去中心化模式 (18.5KB / 門檻 10KB)` (藍色/紫色漸變)
* **即時進度條**：在去中心化模式下上傳檔案時，UI 會出現帶有磨砂玻璃質感（Glassmorphism）的進度彈窗，明確顯示上傳狀態：「*正在將加密資料封裝並安全直傳至 Walrus 節點 (85%)...*」，上傳完畢後無縫切換為「*正在喚起 Sui 錢包進行注資上鏈...*」。

### 2. 一鍵式流暢流程 (Single-Click Flow)
* **背景防遮罩阻塞**：雖然先上傳再上鏈是序列執行的，但前端將上傳過程封裝在提交按鈕的 Loading 狀態內。使用者只需點擊一次「提交問卷/提交答卷」，系統在背景自動完成「加密 $\to$ 檔案直傳 $\to$ 取得 ID $\to$ 喚起錢包」，使用者自始至終只需在網頁上點擊一次「提交」，並在錢包中 Approve 一次，沒有步驟分裂感。
* **上傳失敗與降級 UI**：
  * 若直傳 Walrus 失敗（如超時），UI 會即時中斷並淡入顯示清晰的容錯警告卡片：「*Walrus 網路連線超時，是否切換至備用 IPFS 網路儲存？*」，並提供「重試/切換備用」按鈕。
  * 若後端 BFF 離線導致代付預提功能失效，前端會彈出提示：「*後端代付通道維護中，已自動降級為純去中心化模式，您需要額外簽署一次交易以支付儲存費用*」。

## 合約與鏈上結構變更規劃

### 1. `survey_registry.move` (問卷註冊)
* **`Survey` 物件結構**：
  * `encrypted_content: Option<vector<u8>>` (當 $\le 10\text{ KB}$ 時，直接存放加密問卷明文)
  * `survey_blob_id: Option<vector<u8>>` (當 $> 10\text{ KB}$ 時，存放 Hash/CID)
  * `survey_content_hash: vector<u8>` (固定 32 bytes，不管是哪種模式，都存放明文的 Hash 用於防篡改核對)
  * `storage_fund: Balance<SUI>` (或 `WAL`，發起者預先提撥的答卷儲存代付資金池，僅在問卷大於儲存門檻或預期答卷大於門檻時使用)

### 2. `survey_vault.move` (填答 Claim)
* **`claim` 函數與事件**：
  * 原本：傳入並發射 `encrypted_answers: vector<u8>` (長度隨答案大小增加)
  * 修改後：
    * 支援傳入 `encrypted_answers: Option<vector<u8>>` 或 `answer_blob_id: Option<vector<u8>>`。
    * 當使用 `answer_blob_id` 時，合約內強制校驗該 ID 與 `nullifier` 證明的綁定關係，並從該問卷的 `storage_fund` 中扣除/釋放答卷對應的去中心化儲存費用。
    * 當使用直接儲存時，直接校驗並記錄 `nullifier`。
