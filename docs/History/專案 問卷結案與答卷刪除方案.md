# 專案 問卷結案與答卷刪除方案

## 1. 專案目標與背景
當問卷結案（Creator 關閉 `SurveyVault`）並經過一段冷卻時間後，系統需要提供自動或手動刪除所有問卷及答卷資料的機制。本文件記錄了在當前「Sui 區塊鏈 + 去中心化儲存 (Walrus/IPFS) + BFF 後端快取」架構下，刪除機制的可行性分析、技術路徑對比以及最佳實踐。

---

## 2. 三層儲存結構的刪除可行性

在混合儲存架構中，資料分散在三個層級，其刪除特徵如下：

### A. 區塊鏈鏈上層 (Sui Blockchain)
*   **問卷與金庫物件 (Shared Objects)**：
    *   鏈上存在 `Survey` 物件與 `SurveyVault` 物件。
    *   **刪除可行性**：可行。可透過合約新增 `delete` 函數將物件解構並調用 `object::delete` 銷毀，銷毀時會退還 Storage Rebate 給發起者。
*   **答卷內容與填答歷史 (Claim History)**：
    *   在現行合約中，答卷資料是作為交易參數並經由 `SurveyClaimed` 事件發射。
    *   **限制**：一旦上鏈，寫入區塊鏈歷史的交易輸入與事件 (Events) **永久不可抹除與物理刪除**。

### B. 去中心化儲存層 (Walrus / IPFS)
*   **Walrus 儲存**：
    *   **刪除可行性**：天然支援。上傳資料至 Walrus 時需支付指定 Epoch 數量的租約。若設定在「問卷截止時間 + 保留期限」後租約到期，且不再續費，Walrus 儲存節點會**自動修剪 (Prune) 並物理刪除該 Blob 資料**。
*   **IPFS 儲存**：
    *   **刪除可行性**：若使用託管的 Pinning 服務，可在結案後透過 BFF 調用 API 進行 `unpin`，等待 IPFS 節點垃圾回收自動清除。

### C. 後端快取層 (BFF Cache)
*   **BFF 本地快取**：
    *   BFF 將問卷與答卷密文快取在伺服器硬碟中 (`data/survey_cache` 與 `data/answer_cache`)。
    *   **刪除可行性**：極易實現。可藉由後端定時任務 (Cron Job) 偵測鏈上金庫狀態，若已結案且超過保留期，則主動執行 `fs.promises.unlink` 刪除檔案。

---

## 3. 方案一：現行「事件模式 (Event Mode)」下的刪除與清理

在此模式下，答卷透過事件發射，不佔用最新的鏈上物件狀態。

### 清理機制設計
1.  **後端快取自動清理**：
    *   BFF 啟動定時排程，每日掃描本地快取。
    *   透過 Sui RPC 查詢對應 `SurveyVault` 物件，若 `status == STATUS_CLOSED` 且當前時間大於 `closed_at_ms + 保留時間`，則物理刪除本地對應的問卷與答卷快取檔案。
2.  **對接 Walrus 自動過期**：
    *   上傳問卷/答卷至 Walrus 時，前端根據問卷截止時間動態計算並預付合理的租約期限，到期後讓 Walrus 自動刪除。
3.  **核心密碼學刪除 (Crypto-shredding)**：
    *   問卷與答卷皆在本地加密。當問卷結案且過了保留期限，發起者在客戶端**銷毀其本地保存的解密私鑰**。
    *   **效果**：私鑰銷毀後，即使鏈上或備份節點中仍存有交易歷史密文，在數學上也永久無法被還原，等同於安全銷毀。

---

## 4. 方案二：全新「物件化模式 (Object / DOF Mode)」下的刪除與清理

在此設想下，答卷在填答時被 Mint 成獨立的 `Response` Move 物件，以**動態物件欄位 (Dynamic Object Field, DOF)** 的形式掛載到 `SurveyVault` 底下。

### 清理機制設計
1.  **分批銷毀交易**：
    *   由於單筆交易有 Gas Limit 上限 (50M MIST)，發起者（或 BFF 代付）必須發送**多筆分批交易**（每次 100~200 個物件），調用合約解構 `Response` 物件並呼叫 `object::delete` 進行銷毀。
2.  **存儲費回收 (Storage Rebate)**：
    *   銷毀答卷物件時，Sui 會退還約 **99%** 的存儲費押金（約每份答卷退回 `0.002 ~ 0.003 SUI`）給執行銷毀的帳戶，提供強大的清理激勵。

### ⚠️ 父子物件銷毀的關鍵底層限制
1.  **不可直接刪除父物件**：
    若父物件（金庫/題目卷）下仍掛載有動態欄位，當試圖直接銷毀父物件 `UID` 時，**Sui 合約會直接 Abort 報錯**。必須先將子物件全部移除並銷毀，才能銷毀父物件。
2.  **避免產生孤兒物件 (Orphaned Objects)**：
    若將子物件設定為父物件擁有（Object Owned），直接銷毀父物件後，子物件不會被自動刪除，而是會變成無法存取、無法銷毀的孤兒物件，導致**存儲押金永久鎖死在鏈上**。因此仍必須編寫程式分批手動銷毀。

---

## 5. 方案對比與抉擇建議

| 比較維度                  | 方案一：事件模式 (Event) — *現行*       | 方案二：動態物件欄位模式 (DOF)             |
| :------------------------ | :-------------------------------------- | :----------------------------------------- |
| **最新狀態清理**          | ❌ 無法清理（根本無狀態物件）            | **🟢 可完全物理刪除**（銷毀物件）           |
| **Storage Rebate (退款)** | ❌ 無退款                                | **🟢 可回收大量 SUI 存儲費**                |
| **填答時的 Gas 成本**     | **🟢 極低**（約 0.0015 SUI）             | ❌ 較高（約 0.0055 SUI，增加 3.6 倍）       |
| **清理手續複雜度**        | **🟢 極低**（僅需 BFF 清除快取）         | ❌ 較高（需 BFF 支付 Gas 執行分批銷毀交易） |
| **歷史隱私防範**          | 需依賴密碼學私鑰銷毀 (Crypto-shredding) | 需雙重依賴（銷毀物件 + 密碼學私鑰銷毀）    |

### 決策建議
*   **若重視低成本與高併發體驗**：
    應維持**方案一（事件模式）**。填答時 Gas 消耗極低，代付金庫資金利用率高。資料的刪除應依賴 **「BFF 快取清理 + Walrus 自然過期 + 前端銷毀私鑰 (Crypto-shredding)」** 來實現。
*   **若重視鏈上資料的最新狀態物理清除與存儲代幣回收**：
    可改用**方案二（物件化模式）**。雖然填答成本增加且結案後需用指令分批交易銷毀，但能徹底清空鏈上最新狀態，並回收大量 SUI 押金。
